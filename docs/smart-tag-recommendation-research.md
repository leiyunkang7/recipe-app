## 2. 智能标签推荐方案

### 方案一：基于规则的推荐 (Rule-Based)

根据食材、烹饪方式、口味等自动生成标签

推荐标签类型:
| 类型 | 示例 | 生成规则 |
|------|------|----------|
| 烹饪方式 | 炒、煮、蒸、烤、煎、炸 | 从步骤指令中提取 |
| 口味 | 辣、酸、甜、咸、清淡 | 从调料成分分析 |
| 难度 | 简单、中等、复杂 | 基于步骤数量和时间 |
| 时间 | 快手、慢炖 | 基于总烹饪时间 |
| 季节 | 春季、夏季、秋季、冬季 | 基于当前月份 |
| 场景 | 早餐、午餐、晚餐、甜点 | 基于分类和内容 |

优点: 实现简单，无需训练数据
缺点: 无法发现隐含模式

### 方案二：基于共现的推荐 (Co-occurrence)

频繁一起出现的标签组合，形成标签组

示例:
- (川菜, 辣) 高共现
- (日料, 清淡, 生鱼片) 高共现
- (烘焙, 甜, 面粉) 高共现

优点: 发现潜在标签关联
缺点: 冷启动问题

### 方案三：基于内容的推荐 (Content-Based)

分析食谱的食材、步骤、描述，自动推荐标签

特征提取:
- 食材名称 -> 食材类别标签 (肉类、蔬菜、海鲜等)
- 烹饪动词 -> 方式标签 (炒、煮、蒸等)
- 描述中的形容词 -> 口味标签 (辣、酸、甜等)

优点: 可解释性强，易于实现
缺点: 需要维护规则库

### 方案四：机器学习推荐 (ML-Based)

使用 NLP/ML 模型自动学习标签模式

可选模型:
| 模型 | 优点 | 缺点 |
|------|------|------|
| TF-IDF + 分类器 | 简单高效 | 需要标注数据 |
| Word2Vec/Embedding | 语义理解 | 训练成本 |
| BERT/Transformer | 效果最佳 | 需要 GPU |

优点: 准确性高，可发现复杂模式
缺点: 需要训练数据和计算资源

## 3. 推荐实施方案

### 推荐方案：混合方案（推荐）

**阶段一：规则 + 共现 (快速实现)**
- 实现基于内容的规则推荐
- 实现标签共现分析
- 用户可接受度高的快速方案

**阶段二：ML 增强 (长期优化)**
- 收集用户行为数据
- 训练标签预测模型
- 持续优化推荐效果

### API 设计

```typescript
// POST /api/tags/suggest
interface TagSuggestRequest {
  recipe_id?: string;
  ingredients?: string[];
  title?: string;
  description?: string;
  steps?: string[];
}

interface TagSuggestResponse {
  suggestions: Array<{
    tag: string;
    score: number;
    source: 'rule' | 'cooccurrence' | 'ml';
  }>;
}
```

## 4. 实施计划

### 第一阶段：基础推荐 (1-2 周)
- 创建标签推荐 API 端点
- 实现基于食材的标签推荐规则
- 实现基于烹饪方式的标签提取
- 添加标签管理界面

### 第二阶段：增强推荐 (2-3 周)
- 实现标签共现分析
- 开发标签推荐前端组件
- 用户交互反馈收集
- 推荐效果评估

### 第三阶段：智能优化 (长期)
- 收集用户行为数据
- 训练 ML 标签预测模型
- 个性化推荐优化

---

## 5. 风险与挑战

| 挑战 | 应对方案 |
|------|----------|
| 标签标准化 | 建立标签字典，统一格式 |
| 多语言支持 | 标签国际化处理 |
| 冷启动 | 使用规则+共现混合方案 |
| 用户接受度 | 提供可编辑/删除接口 |

---

## 6. 结论

智能标签推荐系统能显著提升用户体验，建议采用混合方案：

1. 短期: 基于规则和共现的推荐，快速上线
2. 长期: 引入 ML 模型，持续优化

优先实施阶段一，可获得 80% 的效果，仅需 20% 的工作量。
